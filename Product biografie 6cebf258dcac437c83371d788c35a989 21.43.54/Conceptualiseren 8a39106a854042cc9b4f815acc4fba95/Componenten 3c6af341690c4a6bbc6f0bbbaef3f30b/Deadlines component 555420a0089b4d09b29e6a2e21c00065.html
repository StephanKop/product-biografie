Doel van het component

technische flow

Een user voegt een deadline toe via het formulier. De backend verwerkt de postrequest en zet de data door naar de deadline tabel. Deze staat gelinkt met een koppeltabel die ook weer gelinkt is aan de member tabel. Daarna kan zoals bij alle andere componenten de front-end een api call doen naar de backend waardoor de data via een .map() functie kan worden getoond.

Data klaarzetten op de backend

getDeadlines(): Promise<DeadlinesEntity[]> {
        const today = new Date();
        const futureDate = new Date();
        futureDate.setDate(futureDate.getDate() + 100);
        return this.deadlinesRepository.find({where: {deadline: Between(today, futureDate)}, relations: ['members']});
    }

Deze functie haalt alle toekomstige data op uit de repository en wanneer deze functie en returned de data uit de find functie wanneer de getDeadlines() functie gecalled wordt.

Api call naar backend

Hieronder zie je de fetch waarmee ik alle deadlines ophaal.

async function fetchData() {
        const res = await fetch(fetchUrl);
        res
            .json()
            .then((res) => {
                setIsLoaded(true);
                setDeadlines(res);
            })
            .catch(err => setErrors(err));
        return true;
    }

om de promise te resolven heb ik een .then() functie op de response gedaan waarin ik de state Deadlines vul met de response data en stel ik Isloaded state in op true.

fetchUrl is een variabele met de link naar de backend erin die ingesteld wordt met met een prop.

let [fetchUrl, setFetchUrl] = useState(props.apiLink)

Eerste ontwerp

Het eerste ontwerp van het deadlines component bestaat uit deadline balken die korter worden nadat de deadline dichterbij komt. Dit idee komt voort uit onderzoek naar best practices van progress bars (bron: medium.com, best practices in designing awesome progress bars, 2020). Ik vond het een goed idee om de gebruiker te tonen dat de dagen tot de deadline steeds minder worden en daarom de deadline “afloopt”.

Ook zijn de members die geregistreerd staan bij die deadline te zien op de deadline balk. Dit geeft duidelijk aan wie er aan welke deadline werken, zonder teveel tekst te gebruiken. Wanneer een deadline op minder dan 5 dagen afstand is kleurt deze rood om te accentueren dat die deadline aandacht vereist. Wanneer een member afwezig is kleurt de deadline geel om ook dit als aandachtspunt aan te stippen.

2e ontwerp

  • De voornaamste feedback kwam neer op de onduidelijkheid over de aflopende balkjes. Deze heb ik dus veranderd naar balken die zich altijd op 100% van de lengte tonen maar gevuld worden als een progress bar. Zo is het voor de developers duidelijker waar de deadline naar toe loopt omdat de 0% en 100% waarde van de balk duidelijk zijn.
  • Ook heb ik de deadline datum veranderd naar dagen resterend zodat de developers niet meer voor hunzelf hoeven omrekenen hoeveel dagen er dan resterend zijn.
  • Naar wens van de developers heb ik ook een sorteerfunctie toegevoegd waarmee deadlines op naam en datum gesorteerd kunnen worden.
